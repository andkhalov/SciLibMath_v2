## 2. Обнаруженные проблемы

### 2.1 КРИТИЧНО: Centroid R@1 — бесполезная метрика (насыщение)

**Диагноз:** centroid_R@1 ≈ 1.0 для ВСЕХ экспериментов. Метрика насыщена и не различает.

**Причина:** LOO centroid retrieval — query = mean(4 из 5 модальностей), target = mean(все 5). Query и target делят 4/5 компонентов → cosine similarity ≈ 0.95-0.99 → всегда top-1.

**Это НЕ баг кода.** Retrieval работает на полном test set (N=8,122, матрица [8122×8122]). Это дефект дизайна метрики: она измеряет identity preservation, а не семантическое качество.

**Вывод:** Использовать cross-modal R@1 как основную метрику. Centroid R@1 убрать из критерия best model.

### 2.2 Неполная cross-modal матрица: 7 из 20 пар

В `retrieval.py` вычисляются только 7 пар из 20 возможных. Остальные 13 пар (ru→en, ru→lean, latex→en, latex→ru, latex→lean, latex→img, lean→en, lean→ru, lean→img, en→img, ru→img, img→ru) не считаются. Тепловая карта 5×5 наполовину пуста.

**Действие:** Вычислять все 20 пар (или хотя бы сделать симметричными).

### 2.3 Баг: Family B перезаписывает checkpoints Family A

Configs e1b_pairwise.yaml, e2b_pairwise.yaml и т.д. содержат `experiment: e1_pairwise` (без 'b'). Sweep A→B последовательно, B перезаписывает checkpoints A.

**Действие:** Исправить experiment names в B-конфигах (e1b_pairwise, e2b_centroid, ...).

### 2.4 img→* cross-modal ≈ 0 для ВСЕХ экспериментов

Визуальная модальность не вытягивается в cross-modal retrieval. img→en R@1 ≈ 0.001 даже при img→centroid R@1 = 0.93. Модальности кластеризуются вокруг центроида, но img-кластер не совмещён с text-кластерами в cross-modal пространстве.

**Причина:** Разная природа представлений (ResNet features vs SciRus-tiny). AlignNet выравнивает с latex, но не с другими текстовыми модальностями.

## 3. Ключевые находки

### 3.1 E2 (centroid loss) уничтожает cross-modal структуру

E1 (pairwise): en→ru = 0.776, en→lean = 0.511
E2 (centroid): en→ru = **0.008**, en→lean = **0.002**

Centroid InfoNCE без alignment/radial loss схлопывает все модальности в одну точку (D_intra = 1.32, но collapse_score = 0.001 — это ложноотрицательный collapse: модальности различимы по норме, но не по направлению).

### 3.2 E3 (alignment+radial) — ключевой прорыв

E3 восстанавливает cross-modal (en→ru 0.572) при D_intra = 0.06 (в 22 раза лучше E2). Alignment+radial регуляризация — необходимые компоненты для баланса centroid/cross-modal.

### 3.4 E6/E7 (fuzzy/lyapunov) — контроллер "ушёл в угол"

Loss не сходится (11.9 vs 2.5 у E3-E5). Анализ TensorBoard:

```
Финальное состояние λ_t контроллера (E6):
  w_en = 0.10 (MIN), w_ru = 0.10 (MIN), w_lean = 0.10 (MIN)
  w_latex = 0.10 (MIN), w_img = 0.10 (MIN)
  w_g = 5.00 (MAX)
  λ_align = 1.00 (MAX), λ_va = 0.50 (MAX)
  loss_va = 14.59 (доминирует total loss)
```

**Диагноз:** Контроллер загнал все модальные веса в минимум, глобальный вес в максимум, и visual alignment weight в максимум. loss_va = 14.59 доминирует, забивая полезные градиенты.

**Причина:** Линейные консеквенты A_r·s_t (линейные в s_t) + box constraints → система быстро уходит в угол ограничений при alpha=0.01.

## 4. План по тюнингу E6/E7 (Fuzzy + Lyapunov)

### 4.1 Проблемы текущей реализации

1. **Линейные консеквенты в T-S:** u_t = Σ h_r · (A_r · s_t + b_r) — линейная функция s_t → быстро насыщается при box constraints
2. **Alpha=0.01 слишком агрессивно:** A_r масштабируются alpha, но s_t после z-norm имеет значения ~O(1), итого Δλ ~ 0.01-0.1 на шаг при частоте обновления каждый шаг
3. **Box constraints создают "липкие стенки":** λ достигает bounds и застревает, т.к. коррекция u_t монотонна при постоянном сигнале
4. **Правила R1-R6 не имеют обратного действия:** нет правила "если loss снижается — вернуть веса к дефолтам"

### 4.2 Предложения по улучшению

#### Вариант A: Ослабить правила (минимальное изменение)
- **Уменьшить alpha:** 0.01 → 0.001 или 0.0005
- **Добавить decay к дефолту:** λ_{t+1} = Π(λ_t + u_t + γ(λ_default - λ_t)), γ=0.01 — elastic mean-reversion
- **Расширить bounds:** w_m ∈ [0.3, 3.0] вместо [0.1, 5.0] — менее экстремальные значения
- **Обновлять не каждый шаг:** apply controller every K=10 steps

#### Вариант B: Нелинейные консеквенты
- Заменить A_r · s_t + b_r на MLP(s_t) с 1 hidden layer:
  ```
  consequent_r = W2_r · σ(W1_r · s_t + b1_r) + b2_r
  ```
- MLP позволяет нелинейную зависимость коррекции от состояния
- Риск: MLP нужно обучать → нет данных для supervised → нужен RL или meta-learning

#### Вариант C: Вероятностная интерпретация с семплированием (рекомендуемый)
- **Идея:** Вместо детерминированного u_t, интерпретировать h̄_r как вероятности:
  ```
  r* ~ Categorical(h̄_0, ..., h̄_6)    # семплируем одно правило
  u_t = A_{r*} · s_t + b_{r*}           # применяем только его
  ```
- **Или мягкий вариант:** u_t = Σ h̄_r · (A_r · s_t + b_r) + ε, ε ~ N(0, σ²I)
- **Преимущества:**
  - Выход из "липких стенок" через стохастичность
  - Exploration vs exploitation: σ можно уменьшать со временем (annealing)
  - Сохраняется интерпретируемость правил (h̄_r = вероятность активации)
  - Связь с policy gradient (REINFORCE): можно оптимизировать правила
- **Теоретическая связь:** T-S fuzzy ≈ Mixture of Experts; вероятностное семплирование → Bayesian MoE

#### Вариант D: Hybrid (A + C)
1. alpha=0.001 + elastic mean-reversion (Вариант A)
2. Stochastic rule sampling с σ-annealing (Вариант C)
3. Controller step каждые 10 training steps
4. Warmup 200 steps (вместо 50) — не менять λ пока loss не стабилизируется

### 4.3 Тюнинг Lyapunov (E7)

Текущая проблема: V_t penalty добавляется к уже сломанному E6 loss → усугубляет проблему.

**Зависимость:** Сначала починить E6, потом тестировать E7. Lyapunov penalty имеет смысл только при рабочем контроллере.

Идея: V_t как soft constraint, не hard penalty:
```
L_total = L_composite + max(0, ΔV_t - ξ) · penalty_weight
```
Штрафовать только РОСТ V_t сверх порога ξ, а не абсолютное значение.
